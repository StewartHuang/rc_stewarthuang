# AI 使用说明

# AI 工具

OpenCode + Superpowers plugin + DeepSeek V4 Flash

# AI 在哪些关键地方提供了帮助
1. 系统方案设计阶段：
   - 通过 Superpowers Plugin 梳理需求，并生成系统设计文档
   - 在人工对系统方案设计文档时做出的决策和纠正时自动写入到该文档
2. 详细方案+代码实现
   - 由 superpowers 在确定 SPEC 后使用 SubAgent 模式完成首版代码实现
3. 代码测试：生成单元测试代码，并验证结果

# AI 给出过哪些你没有采纳的建议
1. 系统设计环节中在 DB 设计时使用了 外键 和 CHECK 功能
   - 外键在业务实践中带来的问题多于提供的保障
   - CHECK 的逻辑如果放到 DB 层面实现会导致选项变动必须修改表定义，风险较高，适合放到业务层实现
2. 系统设计环节给 API 加上了模板渲染能力
   - 模板渲染是一个非常复杂且需求因业务各异的功能，如果包含在系统内上到生产环境之后需要处理包括变量传入，检验，模板版本管理等复杂逻辑，和投递系统本身的主要职责关系并不紧密
3. 初始设计未明确服务边界
   - AI 提出的设计方案缺少目标与非目标的范围界定，不符合工程最佳实践
   - 人工补充后明确了系统的职责边界，防止设计蔓延

# 关键决策以及原因

## 1. 消息投递可靠性选择

> 消息投递的可靠性要求是什么级别？
> A) 基础投递 — 内存队列 + 失败重试几次，重启后未投递消息会丢失
> B) 持久化投递 — 消息写入本地存储/DB，重启后继续投递，保证不丢
> C) 最终一致/事务性投递 — 消息持久化 + 至少一次投递语义 + 死信队列，需要手动处理长期失败的消息

**最终决策：C) 事务性投递**

当前系统的核心就是投递的可靠性，所以必须要保证投递请求进入系统后一定能被投递出去。持久化存储 + 至少一次投递语义 + 死信队列是必要保障。

## 2. 存储后端选择

> 存储后端用什么？
> A) SQLite — 零依赖，单二进制部署，适合中小规模
> B) PostgreSQL — 需要额外部署，面向高并发/大规模场景
> C) BoltDB/bbolt — 嵌入式 KV 存储，零依赖，读写性能好但查询能力有限

**最终决策：A) SQLite（零依赖）**

当前只需要实现 MVP，为了方便测试和验证，使用单文件的 SQLite 是最佳选择，后续进入生产环境切换到 PG 或 MySQL 都比较容易

## 3. 供应商配置管理方式

> 供应商配置管理方式？
> A) 配置文件 — 启动时加载，修改需重启服务
> B) 管理 API + DB 存储 — 运行时动态添加/修改/禁用供应商配置
> C) 混合模式 — 配置文件启动，也支持运行时 API 管理

**最终决策：C) 混合模式**

配置文件便于版本管理和初始化部署，运行时 API 支持生产环境动态调整而不需要重启服务。

## 4. 系统架构选型

> 架构方案选择？
> A) 单体 Outbox 模式 — 单进程 HTTP API + Background Worker + SQLite
> B) API + Worker 分离 — 独立 API 服务和独立 Worker 进程
> C) 事件驱动 — 引入 NATS/RabbitMQ 作为消息中间件

**最终决策：A) 单体 Outbox 模式**

在MVP和中小规模使用，单二进制部署最简单，功能完整满足需求。分离架构和事件驱动增加了不必要的运维复杂度。

## 5. 目标与非目标（人工补充）

AI 初始设计未包含目标与非目标章节，无法准确定义系统职责边界。人工补充后明确：

**目标：**
- 稳定可靠的外部通知触达服务
- 消息送达策略配置
- 供应商接口管理
- 幂等性投递

**非目标：**
- 不提供外部错误处理 — 外部 API 返回的业务错误由业务系统自行处理
- 不提供请求内容生成及渲染 — 业务系统自行构造投递的 body 内容
- 不提供多请求流程编排 — 专注于单次通知投递

## 6. 回调通知机制

> 投递完成时的回调应该怎么实现？
> A) Worker 内内联执行回调 — goroutine 自包含重试，不影响 Worker 槽位
> B) 独立队列表 + 独立 CallbackWorker 轮询 — 进程重启不丢重试
> C) 同步内联重试 — 回调重试阻塞 Worker 槽位

**最终决策：B) 独立队列表 + 独立 CallbackWorker**

回调不应阻塞正常投递流程。独立队列表（callbacks）+ 独立轮询 Worker 确保进程重启不丢回调重试，且不干扰主 Worker 的投递吞吐。

## 7. HTTP 成功状态码可配置

> 供应商接受哪些 HTTP status code 视为成功？
> A) 列表形式 — `[200, 201, 202]`，需指定具体值
> B) 区间形式 — `"200-399"`，范围匹配
> C) 列表 + 区间结合

**最终决策：A) 列表形式**

列表形式最精确，不同供应商对成功响应有不同定义（如某些供应商仅返回 200 而非 201/204），明确列表减少歧义。默认值 `[200]` 与大多数系统行为一致。
